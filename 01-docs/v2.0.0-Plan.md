# Wordpress Schedule and Checkin

## Version 2.0.0 Summary and High Level Plan

Planning began 2026-04-30.

**Focus:** The overall goal is to provide a data-secure, internally hosted & managed, sign up & attendance platform to utilize in the ICSP community that will help us better plan, track, and report on all events that we host. We want this to be the only tool used for these functions, consolidating on our own platform for quality, functionality and security of data. The timeframe goal is to redeploy in a big way by July 1 or August 1 depending on planning, development, and testing timeframes and availability.

### Changes from current state to v2.0.0 state

#### Features to Add

1. Authentication for user sign ups.
    - Required for consistency of tracking and reports
    - Required for user 'self-management' features
    - A parent must be able to modify their children, so that has to be taken into account as the children won't have their own sign up, but the parent may sign up multiple children.
2. User Self-Management
    - A user needs the ability to modify their sign up in case of cancellation, user errors, or other unforeseen reasons
    - This includes deleting an existing sign up and then add new sign up, no modify existing (over-complicated)
3. Repeat Sign ups (Reoccurring)
    - Specifically for Adoration, allow a user to setup a reoccurring signup (up to 3 months at a time)
    - This should be off by default, but enabled by each event when event is created/modified.
4. Multiple Event Types Template
    - The ability to have a template for Adoration, CF Class Attendance, Fundraisers, and other event types where each type will have their own requirements, functionality, and interfaces.
    - Backend administrators will need to be able to choose the type at event creation
5. Event Generation via JSON
    - CF Class Generation: This feature is actually designed so that we can import data from a JSON file for CF Class generation for dates and rosters
    - Adoration Generation: Generation of dates with number of slots per hour and the hour offerings to populate multiple events at a time
    - Also want to be able to export a current event to JSON that could be used as a template for future events of the same type
6. Manager/Teacher Attendance Tracking
    - If children do not check in using the kiosk (such as CF classrooms), a Teacher/Manager user should be able to take attendance of who is present and who is not.
7. Audit logs
    - Add immutable change history for sign ups, attendance edits, admin changes, and exports to aid troubleshooting and accountability.
8. Background processing
    - Move large imports/exports, JSON generation, and bulk emails to background jobs/queues to avoid timeouts and improve reliability.
    - Provide progress reporting and retry handling for long-running tasks.
9. Granular RBAC
    - Implement role-based permissions (admin, manager, teacher, parent, volunteer) with per-event delegation and fine-grained capabilities.
10. Timezone handling
    - Store event timezones, display localized times to users, and normalize scheduling logic for consistent sign ups across regions.
11. Kiosk resilience & security
    - Support offline/queued check-ins, kiosk-only simplified UI, PIN/QR check-ins, and kiosk lock down modes for minors.
12. Data protection & exports
    - Provide user data export/deletion workflows, consent capture, retention policies, and soft-delete with restore for admin recovery.

#### Changes to Make

1. Ability to control what is shown on the public sign up page for the user
    - This is for protection, default should be first initial and last name, but other options should be available (Full Name, full first name and last initial, etc.) while handling similar names like "Matt Smith" and "Margaret Smith"
    - This needs to be a setting per event
2. Fix email updates to manager/owner to be as often as needed, not hard-coded
3. Complete backend interface update for better efficiency while maintaining features
    - Better reporting interface with simple usage with advanced features.
4. Complete frontend interface updates for each event type
    - Want to be able to handle multiple date sign ups (Adoration) as well as reoccurring

### Version 2.0 Summary

#### Project Goal

To completely update the plugin as it stands today to be a fully-featured and functional application to handle Adoration sign ups, CF youth class attendance, CF Adult Event/Class sign ups and attendance in a simple interface while strengthening our reporting and tracking ability on the backend.

#### This release will address several operational and technical challenges identified in the current plugin

- **Data consistency & identity:** Introduce authentication and parent/child relationships so sign ups are reliably associated with users and can be managed by guardians.
- **Privacy & display control:** Provide per-event options for how attendee names are presented to balance privacy with clarity (full name, initials, last initial, etc.) and handle duplicate/similar names.
- **Scalability & automation:** Add JSON-based event generation and recurring sign ups to automate bulk creation of dates, slots, and rosters.
- **Event-type flexibility:** Support multiple event templates (Adoration, CF classes, fundraisers) with type-specific validation, fields, and UI flows to reduce edge-case complexity.
- **Attendance accuracy:** Combine kiosk check-ins with teacher/manager attendance workflows to ensure presence records are complete and reconcilable.
- **Notification reliability:** Make email cadence configurable and improve batching to reduce missed or overly frequent notifications.
- **Usability & reporting:** Revamp backend interfaces and reporting/export capabilities to provide quick summaries and advanced analysis for admins.

#### Planned functionality highlights

- User authentication, role and relationship management (parents, managers, teachers).
- A self-service interface for users to add, delete, or rebook sign ups and manage dependents.
- Optional recurring sign ups (per-event enablement, configurable duration limits).
- Event templates and per-type configuration with import/export via JSON.
- Bulk event generation for classes and adoration schedules, with safe batch processing.
- Kiosk integration and manager attendance tools, plus reconciliation and reporting tools.
- Per-event privacy/display settings and smarter name-collision handling.
- Configurable notification schedules and improved email delivery processes.
- Backend reporting improvements, CSV/JSON exports, and a modernized admin UI.

#### Compatibility and rollout considerations

- Provide migration paths for existing events and sign ups; keep legacy behavior behind per-event feature flags.
- Design for security and access control around authentication and sensitive data.
- Anticipate performance needs for large imports/recurring generation and mitigate via batching and background jobs.

The goal for v2.0 is to deliver a stable, configurable, and extensible foundation that improves accuracy, privacy, and admin productivity while enabling future features such as calendar sync, payments, and deeper analytics.
